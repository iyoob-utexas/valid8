# valid8 Framework Gap Report

Generated by running a full valid8 test profile against a synthetic sales pipeline.
Run ID: see run_log.md

These are gaps found by following the framework documentation *literally* and noting where it left us without guidance. Each entry includes the specific moment we got stuck and a suggested fix.

---

## Gap 1 -- SLA window is undefined for freshness checks

**Where:** test-grid.md, Source Ingestion > Data Freshness (TMP-001)
**What it says:** "Alert if latest load exceeds the SLA window"
**What it doesn't say:** What the SLA window is, or how to set one for a new project
**Impact:** I used 7 days arbitrarily. A different practitioner would use 24 hours or 30 days. The check means nothing without a defined threshold.
**Suggested fix:** Add a "how to set your freshness SLA" note to the test-grid row and to test-cycle.md Step 2. Even a one-liner: "Default to 24 hours for daily pipelines, 7 days for weekly, one business day after period close for batch."

---

## Gap 2 -- MoM anomaly thresholds use "X%" placeholder throughout

**Where:** test-grid.md, all Anomaly Detection rows
**What it says:** "Over X% change vs the rolling baseline triggers review"
**What it doesn't say:** How to choose X, what a reasonable starting default is, or how to calibrate it when there's no historical data
**Impact:** I used 25% because it felt reasonable. The check fired (correctly) at 116% MoM. But with a new dataset, a practitioner has no basis for choosing a threshold.
**Suggested fix:** Add a threshold-setting guide to docs/process/testing-strategy.md. Minimal version: "Start at 20% for revenue, 30% for volume. Adjust after 3+ periods of clean data."

---

## Gap 3 -- No guidance for Run 1 (no prior period exists)

**Where:** test-cycle.md, Step 3 (Vs Last Pull)
**What it says:** "Compare this load to recent loads for volume, null rates, and pattern changes"
**What it doesn't say:** What to do when there is no prior load to compare to
**Impact:** Step 3 requires `sales_prior.csv`. On a first run, this file doesn't exist. The framework has no "first run" protocol -- no mention of skipping step 3, seeding a baseline, or marking checks as N/A.
**Suggested fix:** Add a "First Run" callout box to test-cycle.md Step 3: "If no prior period exists, mark anomaly checks as BASELINE (not PASS or FAIL). Record current values as the baseline for the next run."

---

## Gap 4 -- Tool guidance is Fabric/Azure-only; no Python/pandas path

**Where:** test-grid.md column "Fabric / Azure Tooling"; docs/process/tool-guidance.md
**What it says:** Every tooling example cites Dataflows Gen2, ADF, Delta Lake, Fabric Notebooks, Power BI
**What it doesn't say:** How to implement any check using Python, pandas, dbt, or Great Expectations
**Impact:** The "Tool-Agnostic Approach" column helps, but only in abstract terms. A practitioner running pandas on a laptop has nothing to reference.
**Suggested fix:** Add a "Python / pandas implementation path" section to tool-guidance.md with one concrete snippet per test category. Doesn't need to be runnable -- just show the pattern (e.g., `df[col].isna().sum()` for null checks).

---

## Gap 5 -- dim_test schema is never defined

**Where:** docs/grid/summary-test-grid.md, "How to build it" step 1
**What it says:** "Export the test metadata from docs/grid/test-grid.md into a dim_test dimension"
**What it doesn't say:** What columns dim_test should have, what format to export to, or what "export" means for a markdown file
**Impact:** I had to invent my own result schema (step, tier, category, test_name, status, detail, dppf_ids) from scratch. A second practitioner would produce something incompatible.
**Suggested fix:** Add a `dim_test` schema table to summary-test-grid.md with columns: test_id, tier, category, test_name, dama_dimension, dppf_ids, threshold, owner. Provide a CSV template.

---

## Gap 6 -- run_id format is never specified

**Where:** docs/grid/summary-test-grid.md, Key Fields
**What it says:** "`run_id` — unique identifier for the run"
**What it doesn't say:** How to generate it, what format to use, whether it should be a UUID, a timestamp, or a sequential integer
**Impact:** I used `RUN-YYYYMMDD-HHMMSS`. Any format would satisfy the spec, making cross-run comparisons unreliable.
**Suggested fix:** Specify a recommended format in summary-test-grid.md. Simplest: `YYYYMMDD-HHMMSS`. Add a note that the format should be consistent across all runs in a project.

---

## Gap 7 -- maximum_possible_score is ambiguous

**Where:** README.md, Scoring Rubric
**What it says:** "`overall_score = (tier1_pass*5 + tier2_pass*2 + tier3_pass*1) / maximum_possible_score`"
**What it doesn't say:** Whether maximum_possible_score is the sum of weights for (a) all tests in the master grid, (b) all tests in scope for this project, or (c) all tests actually executed in this run
**Impact:** The three interpretations produce very different scores. I used option (c) -- executed tests only -- which matches the pass_rate scoping note added earlier. But it's not stated for overall_score.
**Suggested fix:** Add one sentence: "maximum_possible_score is the sum of tier weights for all tests executed in this run -- the same scope as total_tests."

## Gap 8 -- No test dependency / abort rules

**Where:** test-cycle.md, the 8 steps
**What it says:** Describes 8 steps sequentially
**What it doesn't say:** Whether a failure in Step 2 should abort Steps 3-8, or whether all steps always run regardless of prior failures
**Impact:** I ran all 8 steps even though Step 2 found a null key and Step 3 found a massive MoM anomaly. In a real pipeline you might want to stop immediately on a Tier 1 fail.
**Suggested fix:** Add an "Abort policy" to test-cycle.md. Minimal version: "If Step 1 or Step 2 produces a Tier 1 failure, Steps 3-8 may be skipped. Record them as NOT RUN with a reference to the upstream failure."

## Gap 9 -- Referential integrity is Tier 2, but it corrupts final output

**Where:** test-grid.md, Pipeline Processing > Referential Integrity (SEM-001); test-cycle.md remediations
**What happened:** The orphan product_id (P011, not in dim_product) caused a null `category` in the aggregated output. The report shows a category=NaN row -- which would appear in any dashboard built on this data.
**What the framework says:** Referential integrity is Tier 2 (warn and investigate). The framework doesn't address the downstream consequence.
**Impact:** A practitioner following the framework would log a Tier 2 warning and continue -- but the output is silently wrong. The null category row would appear in dashboards.
**Suggested fix:** Add a note to the Referential Integrity row: "Escalate to Tier 1 if orphan records propagate to final output dimensions (e.g., null category in a report)." Also add "referential integrity orphan" to the remediation table in test-cycle.md.

---

## Gap 10 -- No blank result template provided

**Where:** docs/grid/summary-test-grid.md
**What it says:** Defines the output schema and shows one example table
**What it doesn't say:** Provides no starter file (CSV or markdown) a practitioner can copy and fill in
**Impact:** Every team implementing this starts from scratch and produces a different format.
**Suggested fix:** Add two files to docs/grid/: `result-log-template.csv` (one row per test, all columns from dim_test + status + detail) and `summary-template.md` (the scorecard table pre-populated with headers only).

---

## Gap 11 -- WARN vs FAIL is undefined for Tier 2

**Where:** framework/README.md severity tiers; test-cycle.md failure behavior
**What it says:** "Tier 2 / Important: warn and investigate. The run may continue under review, with quarantine actions for rows or alerts."
**What it doesn't say:** When a Tier 2 result should be recorded as FAIL vs WARN -- the summary grid shows status as "pass / fail / warning" but the criteria for each at Tier 2 is not defined
**Impact:** The referential integrity check and the output null category both produce different severity of outcome but I had to decide myself whether each was WARN or FAIL.
**Suggested fix:** Add one line to the tier definitions: "Tier 2 results are FAIL when the pipeline action is quarantine or block, and WARN when the action is alert or review."

---

## Gap 12 -- Future-dated records not addressed

**Where:** test-grid.md, Source Ingestion > Data Freshness (TMP-001)
**What it says:** Checks that data arrived within the SLA window (data is not *too old*)
**What it doesn't say:** What to do when records are dated in the *future* (a common issue with batch pipelines that backfill or with incorrect source timestamps)
**Impact:** My test data included dates up to 2026-06-30 (end of month). The freshness check passed because I was checking for staleness, not for future-dated records. Future dates would corrupt any time-based aggregation.
**Suggested fix:** Add a separate check to the Source Ingestion category: "Future-Date Check -- no records dated beyond the current load date." This maps to STR-009 (numeric range bounds) applied to the date field.

## What the framework does well

- The 8-step test cycle is easy to follow -- each step has a clear pass condition
- DPPF IDs make it easy to trace from the grid row to the detailed test catalog
- The scoring rubric (READY/REVIEW/BLOCKED) is clear and immediately usable
- The remediation table is a good pattern -- it just needs more entries
- The three-tier model maps cleanly to Python conditionals (pass/warn/fail)
